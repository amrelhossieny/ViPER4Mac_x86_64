## Where We Started
macOS audio DSP app showing "Driver Not Found" / "Status: Not Active". No audio flow at all.

## All Fixes Applied So Far ✅

### Fix 1 — Wrong Plugin Type UUID
Info.plist had the wrong CFPlugInTypes UUID. Fixed to `443ABAB8-E7B3-491A-B985-BEB9187030DB`
(discovered by diffing against AirPlay.driver).

### Fix 2 — XPC vs In-Process Mode
Removed `AudioServerPlugIn_XPCServiceName` key. Driver now loads in-process via dlopen,
same as AirPlay.driver. No XPC service needed.

### Fix 3 — Architecture Mismatch
Leftover arm64 XPC binary was embedded in an x86_64 build. Fixed by full `make clean`
and correct arch flags throughout.

### Fix 4 — Xcode Overwrites Info.plist
Xcode's builtin-infoPlistUtility was replacing our custom plist on every build,
stripping CFPlugInFactories and CFPlugInTypes. Fixed by adding a `fix-plist` Makefile
step (Python3) that re-patches the plist after Xcode builds it, then re-codesigns.

### Fix 5 — Shared Memory Permissions
Driver (runs as _coreaudiod) was creating /tmp/viper4mac_ring.bin with restrictive
permissions that the user app couldn't write to. Fixed by adding fchmod(fd, 0666)
after open() in initSharedMemory(). Verified with ls -la showing -rw-rw-rw-.

### Fix 6 — Swift Can't See ViPERSharedRing C Struct
Swift target couldn't import the C header due to _Atomic types being incompatible.
Fixed in ViPERSharedRing.h by using preprocessor guards:
  - C++  sees: std::atomic<uint64_t>
  - C    sees: _Atomic uint64_t
  - Swift sees: uint64_t (same memory layout, zero-overhead)
Added ViPERSharedRing.h to the bridging header.

### Fix 7 — findRealOutputDevice() Returning nil
With ViPER set as default output, findRealOutputDevice() was returning nil because it
only checked the current default (which was ViPER). Fixed to iterate all devices and
return the first alive non-ViPER device with output streams.

### Fix 8 — Volume Control Not Working
Built-in Intel Mac audio doesn't accept kAudioDevicePropertyVolumeScalar writes on
the main element (AppleHDAEngineOutput). Pushing the virtual device's volume to the
real device always failed silently.
Fix: Replaced hardware volume push with software gain in Swift.
  - syncVolumeToOutput() now caches vol/mute values into currentVolume/currentlyMuted
  - inputIOProcCallback applies gain multiplier to samples after DSP, before writeToRing
  - Volume slider now works correctly ✅

### Fix 9 — Ring Buffer Drop-on-Full
Old writeToRing used drop-oldest which caused audible tearing when the ring filled.
Replaced with drop-on-full: if there's no space, skip the entire write rather than
corrupt mid-buffer data.

### Fix 10 — Sample Rate Mismatch (driver vs real device)
Driver started at 48000 Hz, real device runs at 44100 Hz.
Fixed with two parts:
  (a) Driver: overrode SetNominalSampleRateImpl() to propagate rate changes to all
      stream physical/virtual formats and update ZeroTimeStampPeriod. Added
      SetAvailableSampleRatesAsync (44100, 48000, 88200, 96000) AFTER stream creation.
  (b) Swift: matchSampleRates() pushes the real output device's rate to the virtual
      device on startup, then polls until the virtual device confirms the new rate
      (up to 500ms timeout).
Result: Virtual device now runs at 44100 Hz, confirmed in driver.log every startup.

### Fix 11 — Structured Driver Logging
Replaced raw fprintf/fclose pattern with a proper DriverLogger class matching the
Swift FileLogger style exactly:
  - Timestamp format: "yyyy-MM-dd HH:mm:ss.SSS [Category][LEVEL] message"
  - 2MB log rotation to driver.old.log
  - Thread-safe via std::mutex
  - LOG_DEBUG / LOG_INFO / LOG_WARN / LOG_ERROR macros
  - Comprehensive coverage: SHM init, rate changes, overflows, underruns
File: ViPERDriver/ViPERDriver.cpp

### Fix 12 — TCC Permissions Blocking Audio Flow
coreaudiod was zeroing out all mixed audio before passing it to the driver because
the app was not in the System Audio Recording / Screen Recording privacy list.
Symptom: WriteMixed max=0.000000 for all 25,000+ callbacks even with audio playing.
Fix:
  (a) Add ViPER4Mac.app to System Settings → Privacy → Screen & System Audio Recording
  (b) Full system reboot required (killall coreaudiod alone is not enough for TCC)
  (c) Added NSMicrophoneUsageDescription to ViPER4Mac/Info.plist
Result: After reboot, max values go non-zero immediately on audio playback. ✅

### Fix 13 — AudioConverter Removed for Same-Rate Path
AudioConverter was FORCED ACTIVE even when both devices ran at 44100 Hz.
This added unnecessary latency and a second layer of buffering complexity.
Fix: setupAudioConverter() now returns early (isConverterActive = false) when
inputRate and outputRate differ by less than 1 Hz. Direct ring path is used.
Result: Simpler signal path, lower latency, no extra buffering layer.

### Fix 14 — IOProc Consuming Entire SHM Ring in One Shot
IOProc was reading min(availFrames, 4096) per callback. If SHM had accumulated
~31744 samples (>4096 frames), the IOProc consumed all 4096 frames in one tick,
stuffed the Swift ring, and the output callback could only drain 512 per tick.
Result was ring overflow → dropped buffers → glitches.
Fix:
  (a) Steady-state: cap consumption at halBufferFrames (512) per IOProc tick
  (b) Catch-up mode (SHM > 85% full): drain aggressively up to 4096 frames
  (c) Ring throttle (ring > 80% full): skip the IOProc tick entirely; data
      stays safely in SHM which is 4× larger (32768 samples capacity)

### Fix 15 — SHM Ring Full at Startup (Stale Data Backlog)
The driver starts writing to SHM as soon as coreaudiod loads it. The Swift app
doesn't map SHM until several seconds later. By startup time, SHM was already
97% full (31744/32768 samples), causing 1000+ overflow events in 45 seconds.
Fix: On mapSharedMemory(), reset readPos = writePos (atomic store), discarding
all stale pre-startup data. The app then reads only fresh samples from the moment
the IOProc starts.
Result: No more startup overflow cascade. shmUsed drops from 31744 to ~1114
immediately after startup. ✅

## Remaining Issues ❌

### Issue 1 — Audio Plays Slower Than Normal Speed (CRITICAL)

**Symptom:**
Audio sounds slightly slower/lower-pitched than real speed when processing is enabled.
Disabling processing restores normal speed, re-enabling brings back the slowdown.

**Root cause (suspected):**
The virtual device and the real output device are two independent hardware clocks.
Even though both are set to 44100 Hz nominally, their actual oscillator frequencies
differ by a tiny amount (clock drift). Over time this causes the ring buffer to
either slowly fill (if virtual runs slightly faster → slowdown) or drain (underrun).

Current architecture has NO clock domain bridging between the two devices.
The IOProc writes at the virtual device's clock rate; the AUHAL output reads at
the real device's clock rate. These rates diverge by tens of ppm over time.

**What this looks like in logs:**
- shmUsed oscillates between ~9000-23000 (not stable) — indicates rate mismatch
- The driver delivers 278-279 frames alternating (not a clean 278.0) — HAL is
  compensating for the drift internally but we're not

**Proposed fix:**
Implement AudioConverter-based ASRC (Asynchronous Sample Rate Conversion) between
the IOProc and the AUHAL output. Even at "44100→44100", the AudioConverter with
kAudioConverterSampleRateConverterQuality_Max acts as a rubber-band that absorbs
clock drift transparently. This was previously attempted but disabled because it
was FORCED ACTIVE even at matching rates (causing extra latency). The correct
approach is to keep it active but configure it properly so it only engages when
drift is detected, not as a constant conversion step.

Alternative: Use a simple feedback loop — if ringAvail drifts above or below a
target (e.g. 4096 frames), nudge the read rate by ±1 sample per callback to
compensate without introducing the AudioConverter overhead.

### Issue 2 — Occasional Glitches / Pops (MODERATE)

**Symptom:**
Infrequent audio pops, reduced from constant crackling but not eliminated.

**Root cause (suspected):**
Same clock drift as Issue 1. When the ring buffer drains to near-empty, the
output callback's readFromRing() hits the underflow branch and zero-fills
some samples → audible silence gap → pop when audio resumes.

Also possible: the 80%/85% thresholds in the IOProc flow control cause brief
over/under-production phases that create small discontinuities.

**Evidence from logs:**
- shmUsed varies between 9544 and 23250 over ~2 minutes (not stable)
- ringAvail stable at 4895 for 1 minute (good) then occasional drops

**Proposed fix:**
Same as Issue 1 (ASRC). Once clock drift is absorbed, the ring level will be
stable and the underflow branch in readFromRing will never trigger.

## Next Steps

### Step 1 — Implement proper ASRC to fix clock drift
Re-enable AudioConverter but correctly this time:
  - Only activate when both devices have the same nominal rate (same-rate ASRC)
  - Configure with kAudioConverterQuality_Max for smooth resampling
  - The converter's job is NOT rate conversion but clock domain bridging —
    it will consume slightly more or fewer samples per output callback to
    keep the ring buffer level stable
  - Key insight: AudioConverterFillComplexBuffer pulls data from the
    converterDataProc; by controlling how much we provide, we naturally
    regulate the ring level

### Step 2 — Or: implement a simple drift correction without AudioConverter
If ASRC is too complex/introduces latency, implement a lightweight feedback loop:
  - Target ring level: 4096 frames (50% of 8192 capacity)
  - If ringAvail > 4096+256: skip one IOProc tick (output drains faster)
  - If ringAvail < 4096-256: consume an extra 64 frames from SHM this tick
  - This creates a soft PLL that corrects clock drift without sample-rate math

### Step 3 — Fine-tune IOProc thresholds after Step 1 or 2
Once clock drift is handled, re-examine the 80%/85% thresholds to see if
they can be relaxed or removed entirely.

### Fix 16 — Audio Plays Slower Than Normal Speed (CRITICAL — RESOLVED)

**Root Cause:**
The driver was starting at 48000 Hz (kViPERDefaultSampleRate). The HAL schedules
its entire I/O timeline at driver creation time based on ZeroTimeStampPeriod.
Even after SetNominalSampleRate(44100) was called successfully, the HAL's internal
mix engine was already locked to the 48000 Hz cadence, delivering 278 frames per
callback instead of 512.

The result: driver wrote ~47,800 frames/sec into SHM, Swift output consumed
~44,100 frames/sec → audio played back at ~94% speed (slow/lower-pitched).

**Fix:**
Changed driver initial rate from 48000 → 44100 Hz in ViPER4Mac_Create():
  const Float64 initialRate = 44100.0;  // was kViPERDefaultSampleRate (48000)
  deviceParams.ZeroTimeStampPeriod = (UInt32)initialRate;  // 44100

Removed matchSampleRates() from AudioEngine.swift — no rate adjustment needed
since driver and real device both start at 44100 Hz.

Added ring-level PLL in IOProc for tiny residual clock drift (±1 frame per
100 callbacks) — handles the ~10-100 ppm oscillator difference between the
virtual device and real hardware without any audible pitch change.

**Result:**
  - frames=512 per callback (correct for 44100 Hz)
  - shmUsed=512 (stable, not growing)
  - ringPct=40% (healthy, PLL maintaining target)
  - No pitch shift, no glitches, no slowdown ✅

### Fix 17 — Removed Unnecessary Audio Converter for Same-Rate Path (CLEANUP)

AudioConverter was previously re-enabled "for clock drift" but was broken:
  1. Passthrough closure never set ioNumberDataPackets → 0 output frames
  2. Rate adjustment math was inverted (decreased rate when ring was empty)
  3. Drove effectiveOutputRate to 43879 Hz → complete silence

Fixed by reverting to direct ring write for same-rate path.
AudioConverter is now ONLY active when rates actually differ (future use).
Clock drift handled by PLL instead — simpler, lower latency, correct.
