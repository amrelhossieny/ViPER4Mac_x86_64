# Updated Progress Report

## What We Started With
A macOS audio DSP app showing "Driver Not Found" and "Status: Not Active".

---

## Session Timeline & What We Fixed

### ❌ Problem 1 — Wrong Plugin Type UUID
**Root cause:** `Info.plist` had `293CCA2B-7539-41BE-8CFD-8F6229060E3F` instead of the correct `kAudioServerPlugInTypeUUID`.

**Fix:** Changed to `443ABAB8-E7B3-491A-B985-BEB9187030DB` (discovered by comparing with AirPlay.driver).

---

### ❌ Problem 2 — XPC Mode vs Bundle Mode Confusion
**Root cause:** Adding `AudioServerPlugIn_XPCServiceName` key forced XPC mode, but our XPC service didn't implement the correct protocol. Removing it caused the driver to be ignored entirely.

**Fix:** Removed the XPC key entirely. Driver now loads **in-process** like AirPlay — `dlopen` directly into coreaudiod.

---

### ❌ Problem 3 — Architecture Mismatch (arm64 cached in x86_64 build)
**Root cause:** A leftover arm64 XPC binary from architecture experiments got embedded into the driver bundle.

**Fix:** Full `make clean` + proper arch flags everywhere.

---

### ❌ Problem 4 — Xcode Overwrites Info.plist
**Root cause:** Xcode's `builtin-infoPlistUtility` replaces our custom plist during build, stripping `CFPlugInFactories`, `CFPlugInTypes` and adding wrong keys.

**Fix:** Added `fix-plist` step in Makefile using Python3 to patch the plist after Xcode builds it, then re-sign.

## Root Cause of Remaining Issue

### Problem A — Shared Memory Not Created
The driver's `initSharedMemory()` runs inside `ViPERIOHandler` constructor:
```cpp
// ViPERDriver.cpp
ViPERIOHandler() {
    TPCircularBufferInit(&ringBuffer_, ...);
    initSharedMemory();  // ← creates /com.viper4mac.shm
}
```

But `ViPERIOHandler` is created inside `ViPER4Mac_Create()` which runs as `_coreaudiod` user. The SHM should be created — but the app can't find it. Likely a **permissions or timing issue**.

### Problem B — AudioEngine Never Starts
Looking at `AudioEngine.start()`:
```swift
// AudioEngine.swift
func start() {
    // Waits up to 10 seconds for virtual device
    var viperDevice = findViPERDevice()  // ← this should now work ✅

    // Gets real output device
    guard let realDevice = findRealOutputDevice()  // ← needs checking

    // Maps shared memory
    mapSharedMemory()  // ← shm_open("/com.viper4mac.shm") fails ❌

    // Sets up IOProc
    setupInputIOProc()

    // Sets up output AUHAL
    setupOutputUnit()

    // Sets ViPER as default output
    setDefaultOutputDevice(inputDeviceID)
}
```

The engine may be failing at `mapSharedMemory()` and bailing out silently.

### Step 3 — Fix SHM Permissions
The driver runs as `_coreaudiod`, app runs as user `eren`. The SHM may have wrong permissions:
```cpp
// In initSharedMemory() - make world readable/writable
int fd = shm_open(VIPER_SHM_FILE, O_CREAT | O_RDWR, 0666);
fchmod(fd, 0666);  // ← add this explicitly
```

### Step 4 — Fix findRealOutputDevice()
With ViPER now set as default output, `findRealOutputDevice()` may return `nil` because:
```swift
private func findRealOutputDevice() -> AudioDeviceID? {
    let currentDefault = getDefaultOutputDevice()
    // currentDefault IS ViPER now — so this returns nil!
    if getDeviceUID(currentDefault) != viperUID ...
}
```

The engine needs to find the **real speakers** before it sets ViPER as default.

### Step 5 — Fix Engine Start Order
```swift
// Current (wrong on first run):
// 1. Find ViPER device ✅
// 2. Get default output → returns ViPER (wrong!) ❌
// 3. Map SHM
// 4. Start IOProc

// Fixed order:
// 1. Find ViPER device
// 2. Save current default BEFORE installing ViPER
// 3. Find real output from saved default
// 4. Map SHM
// 5. Start IOProc
// 6. Set ViPER as default
```
